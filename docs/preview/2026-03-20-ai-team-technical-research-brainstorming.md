# AI 团队技术调研脑暴文档（初稿）

- 日期：2026-03-20
- 角色：general strategist
- 类型：技术调研沉淀（GitHub/官方文档/公开研究）
- 目标：为当前 AI 团队流程优化提供外部对标与可执行方向

---

## 1) 问题陈述与目标

### 问题陈述
当前流程存在以下痛点：
1. 虽然已有脑暴文档与 QA 拆分，但员工仍拿到过大上下文，导致执行偏差明显。
2. 流程中 TDD + QA 验收 + 测试复验存在重复验证，造成验收阶段时间与 token 消耗偏高。
3. 组织方式偏“前后端角色分工”，模块责任边界不够清晰，交接损耗高。

### 目标
1. 降低上下文体积与任务偏差。
2. 降低重复验收带来的 token/时间浪费。
3. 建立“按模块推进、分段验收、快速闭环”的团队执行机制。

---

## 2) 约束与假设

### 约束
- 不改变整体“多智能体协作 + 验收保障”方向。
- 需要兼顾质量，不可单纯以速度换准确性。
- 方案需可逐步落地，不依赖一次性大重构。

### 假设
- 当前平台支持任务拆分、角色调度、阶段验收。
- 可以引入“模块组长/经理代理”做中间验收闸门。
- 可以对上下文传递策略做结构化收敛（摘要、契约、工件引用）。

---

## 3) 外部技术调研（对标摘要）

### 3.1 任务拆分：单任务单目标成为共识
主流框架强调“每个任务只做一件事 + 明确 expected output/contract”，避免将多个复杂动作塞进一个大任务。

**启示**：
- 任务最小化不是“写得更细”，而是“输入/输出边界可验证”。
- 拆分后可在更早阶段暴露偏差，降低后期返工成本。

### 3.2 编排方式：分层优于平铺
普遍实践是“Manager/Planner -> Specialists -> Reviewer/Runner”的分层结构，而不是所有角色共享大上下文并行执行。

**启示**：
- 让“谁负责拆分与收敛”变成显式职责。
- 通过经理/组长层吸收复杂性，降低执行层上下文负担。

### 3.3 上下文控制：按需注入，不传全量历史
多框架强调仅传递必要上下文（结构化摘要 + 关键工件），并通过状态持久化/线程记忆支持恢复。

**启示**：
- 传递“决策摘要 + 任务契约 + 上游产物链接”，而不是整段聊天记录。
- 配合 checkpoint/interrupt，失败时局部回退而非全链路重跑。

### 3.4 验收策略：前移并分层，减少重复验证
常见有效形态是三层门禁：
1. Worker 自检（最小可行验证）
2. 模块内验收（组长对契约一致性把关）
3. 跨模块集成验收（轻量 smoke）

**启示**：
- QA 不应对所有任务重复全量回归。
- 验收应与任务粒度同频，模块内先收敛，再做系统级确认。

### 3.5 故障恢复：卡住检测与可恢复执行
一些系统提供循环检测/暂停恢复机制，支持在问题点回退与继续，避免重复消耗。

**启示**：
- 引入“无进展检测 + 人工决策点 + 局部重试”可显著降低浪费。

## 4) 候选方向（高层）

### 方向 A：效率优先（最小 token 方案）
- 强制超小任务粒度（短周期提交 + 快速验收）
- QA 只抽检关键点，更多验收前移到模块组长
- 集成测试频率降低到里程碑级

**优点**：成本下降快、吞吐提升明显
**风险**：若组长把关不足，系统性缺陷可能后移暴露

### 方向 B：质量优先（稳健方案）
- 保留较多验证层级，但每层验证范围明确去重
- 模块组长验收 + QA 重点场景验收 + 测试最终兜底
- 强约束契约与回归基线

**优点**：质量稳定、回归风险低
**风险**：速度与 token 优化幅度有限

### 方向 C：平衡方案（推荐）
- 模块化拆分 + 组长负责模块闸门
- Worker 自检成为硬门槛；QA 从“全量复验”改为“风险驱动验收”
- 系统测试改为“增量触发 + 里程碑兜底”

**优点**：在可控风险下，兼顾成本与质量
**风险**：需要先建立模块边界与验收契约模板

---

## 5) 推荐设计（本轮建议）

推荐采用 **方向 C：平衡方案**，并分阶段推进：

### 阶段 1：组织重构（轻量）
- 从“前后端角色主导”转为“模块主导 + 模块组长负责制”。
- 每个模块定义：范围、输入、输出、验收标准、依赖接口。

### 阶段 2：任务契约化
- 每个任务必须有：
  - 输入（必要上下文）
  - 输出（工件格式/验收条件）
  - 失败处理（回滚点/重试策略）
- 未满足契约不得进入下游。

### 阶段 3：验收去重
- Worker 自检：必须通过最小测试与静态检查后才可提交。
- 模块组长：只验“模块契约一致性 + 接口影响”。
- QA：聚焦跨模块高风险路径，不再逐任务重复验收。
- 测试：系统级回归按里程碑触发，而非每个微任务全跑。

### 阶段 4：上下文治理
- 默认只传递“结构化摘要包”：
  - 任务目标
  - 当前约束
  - 上游输出摘要
  - 必要工件引用
- 超过阈值自动触发压缩/总结。

### 阶段 5：故障恢复
- 增加“无进展检测”与“人工决策点”机制。
- 支持在模块边界回退重试，避免全链路重做。

---

## 6) 风险与缓解

1. **模块边界划分不清**
   - 缓解：先从 1~2 个核心模块试点，沉淀边界模板。

2. **组长负荷过高**
   - 缓解：限制组长验收关注面（只看契约与接口），细节留给 worker 自检。

3. **QA 对新职责切换不适应**
   - 缓解：明确 QA 从“全量复验”转为“风险驱动验收”的规则与样例。

4. **过早压缩上下文导致信息丢失**
   - 缓解：摘要包保留“原始工件引用”，支持按需回看。

## 7) 验证策略（面向流程优化）

建议采用 3 组指标持续评估：

1. **效率指标**
- 单任务平均验收时长
- 单任务平均 token 消耗
- 端到端交付周期

2. **质量指标**
- 验收一次通过率
- 模块内返工率
- 集成阶段缺陷发现率

3. **协作指标**
- 上下文包平均体积
- 任务偏差率（目标与交付不一致）
- Blocker 平均解决时长

以 2~3 周为一个评估周期，对比“改造前 vs 改造后”。

1. 模块边界先按“业务域”还是“技术域”划分？
2. 组长是否由固定角色担任，还是按模块轮值？
3. QA 风险分级标准如何定义（P0/P1/P2）？
4. 你的系统里“上下文阈值”应按 token、消息轮数，还是工件数量触发？

## 9) 结论（本轮确认版）

调研结果与当前讨论收敛为以下执行结论：
- 方向保持“细化任务 + 模块化验收 + 模块组长负责”，并按 TDD 思路前置验收标准。
- 流程调整为：需求入口 → 模块拆解 → 模块验收标准定义 → Worker 开发自检交付 → 模块验收 → 集成冒烟 → 全局人工验收。
- 验收策略采用“模块验收标准化，全局验收人工化”：
  - 模块层：作为主门禁，按明确标准执行，保证一致性。
  - 全局层：由人工做综合验收判断，避免重复自动化/流程化验证造成的无效时间消耗。
- 当前阶段以准确性优先，不推进纯自动化全局验收；待准确性指标稳定后，再评估自动化升级。
- 先做小范围试点，用效率与质量指标验证后再推广。
