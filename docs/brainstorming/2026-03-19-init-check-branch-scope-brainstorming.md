# 初始化检查分支范围优化 Brainstorming

- 角色：general strategist
- 日期：2026-03-19

## 问题陈述与目标

当前初始化检查集中在 `cmd/root.go` 的 `PersistentPreRunE`，除少数白名单命令外，几乎所有命令都会先检查 `.agents/rules/` 是否存在。现有行为会导致 worker 使用场景也被要求先完成项目初始化。

本次目标是将规则收窄为：

- `main` 分支执行时，必须完成初始化。
- worker 分支执行时，不要求初始化。
- 保持现有交互式/非交互式初始化提示逻辑不变。
- 以最小改动完成，不扩散到多个命令实现中。

## 约束与假设

### 约束
- 只做最小必要改动，不重构命令架构。
- 不改变 `agent-team init`、`worker create`、`worker open` 等命令的职责边界。
- 不引入新的配置项或环境变量。
- 保持现有非交互模式报错逻辑不变。

### 假设
- worker 分支命名约定稳定为 `team/<worker-id>`。
- 当前分支可通过 `internal/git.go` 的 `CurrentBranch()` 获取。
- 当前需求按已批准方案 A 执行：仅 `team/*` 放行，其他非 `main` 分支仍保持现状。

## 候选方案与权衡

### 方案 A（推荐）
在 `cmd/root.go` 的统一初始化检查入口增加分支判断：

- `main`：必须初始化
- `team/*`：跳过初始化检查
- 其他分支：保持当前行为，仍要求初始化

**优点**
- 改动最小，逻辑集中。
- 与现有 worker 分支命名规则直接对齐。
- 不需要改动各子命令。

**缺点**
- 分支规则直接写在 root gate 中，后续若再扩展例外分支，需要继续调整该入口。

### 方案 B
抽出独立 helper，如 `RequiresInitialization(branch string) bool`，由 `cmd/root.go` 调用。

**优点**
- 可测试性更好。
- 规则表达更清晰，便于后续扩展。

**缺点**
- 比当前需求所需改动更大。
- 对本次“先优化一个问题”的目标来说偏重。

### 方案 C
基于“当前是否位于 worker worktree”而不是分支名来决定是否跳过初始化检查。

**优点**
- 语义更接近“worker 使用时不需要初始化”。

**缺点**
- 判定复杂度更高。
- 容易引入边界问题，且没有直接复用已有稳定约定（`team/*`）。

## 推荐设计

采用 **方案 A**。

### 原因
- 已有 worker 分支命名就是 `team/<worker-id>`，无需新增识别机制。
- 初始化检查当前已经集中在 `cmd/root.go`，适合做单点修正。
- 能在不影响其他命令职责的前提下实现需求。

## 架构设计

### 入口
继续使用 `cmd/root.go` 的 `PersistentPreRunE` 作为唯一初始化检查入口。

### 判定逻辑
在执行 `.agents/rules/` 检查前：

1. 获取当前 git 分支。
2. 根据分支决定是否需要初始化检查：
   - `main` → 需要
   - 前缀为 `team/` → 不需要
   - 其他分支 → 需要
3. 仅当“需要初始化”时，才执行现有 `.agents/rules/` 存在性检查与交互提示。

## 组件影响

### 需要修改
- `cmd/root.go`
- `cmd/root_init_test.go`

### 不需要修改
- `cmd/worker_create.go`
- `cmd/worker_open.go`
- `internal/git.go`（现有 `CurrentBranch()` 已足够）

## 数据流

1. 命令进入 `PersistentPreRunE`。
2. 完成 git repo 检测并构造 `GitClient`。
3. 获取当前分支名。
4. 分支策略判断是否需要初始化检查。
5. 若需要，再检查 `.agents/rules/` 是否存在。
6. 若缺失，则沿用现有：
   - 非交互模式直接报错
   - 交互模式提示是否执行 `agent-team init`

## 错误处理

- 若分支读取失败，当前 `CurrentBranch()` 会保守回退为 `main`。
- 该行为意味着失败时仍执行初始化检查，符合“宁可严格、不要误放行”的方向。
- 初始化相关提示文案与错误文案可保持不变，避免不必要的用户界面变更。

## 验证与测试策略

需要覆盖以下场景：

- `main` 分支且未初始化 → 返回初始化错误
- `team/test-001` 分支且未初始化 → 放行
- `feature/foo` 分支且未初始化 → 仍返回初始化错误
- 已初始化项目 → 保持通过

测试重点：
- 验证 root gate 的分支条件是否正确生效
- 验证放行仅限 `team/*`
- 验证现有初始化成功路径未被破坏

## 风险与缓解

### 风险 1：误将其他分支也放行
**Why:** 如果条件写宽，例如“非 main 全放行”，会偏离已批准规则。
**How to apply:** 明确只匹配 `team/` 前缀，不做“非 main”兜底放行。

### 风险 2：测试无法稳定构造目标分支
**Why:** 现有测试主要覆盖初始化目录存在与否，未按分支分层。
**How to apply:** 在临时 git 仓库内显式创建并切换到 `main` / `team/*` / `feature/*` 分支后再调用 `PersistentPreRunE`。

### 风险 3：后续需求演变为“仅 main 要求初始化”
**Why:** 本次用户明确批准的是更窄规则：仅 worker 放行。
**How to apply:** 先按当前批准实现，若后续调整，再抽 helper 做规则升级。

## 开放问题

无。当前规则边界已由用户确认：

- 采用方案 A
- 仅 `team/*` worker 分支放行
- 其他分支保持当前初始化要求
