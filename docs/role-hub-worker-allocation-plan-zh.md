# role-hub 可执行任务拆分与 Worker 分配方案（中文）

## 1) 前端/后端可并行模块

| 模块ID | 模块名称 | 类型 | 优先级 | 建议并行阶段 | 上游输入 | 核心输出 | 完成判定（DoD） |
|---|---|---|---|---|---|---|---|
| M0 | 接口与数据契约冻结（API + DB + 状态机） | 跨端 | P0 | S0（先行） | PRD、design.md | OpenAPI 草案、表结构草案、状态流转图 | 评审通过且冻结版本号 `v1` |
| M1 | CLI 异步上报（`find` 后 fire-and-forget） | 后端/CLI | P0 | S1（并行） | M0 契约 | CLI 异步上报实现、失败不阻塞策略 | `find` 耗时无显著回归；上报失败不影响命令成功 |
| M2 | Ingest API（验签+时间窗+幂等） | 后端 | P0 | S1（并行） | M0 契约 | `/ingest` API、幂等 upsert、审计日志 | HMAC 校验通过；重复请求不重复写入 |
| M3 | Postgres 模型与索引（`role_records/ingest_events/sync_jobs`） | 后端 | P0 | S1（并行） | M0 契约 | 迁移脚本、索引、约束 | 迁移可回滚；核心查询具备索引命中 |
| M4 | Normalize Worker（校验+状态流转+重试） | 后端 | P0 | S2（并行） | M2、M3 | 标准化任务、`discovered→verified/invalid/unreachable` | 可重试、可观测、状态正确 |
| M5 | Query API（目录列表/搜索/筛选/详情/仓库页） | 后端 | P0 | S2（并行） | M3（可先用样本数据） | 读接口与分页筛选能力 | 默认仅返回 `verified`；P95 达标 |
| M6 | Public Web（首页/目录/角色详情/仓库详情） | 前端 | P0 | S2（并行） | M0 + M5 Mock/真实接口 | 4 个 P0 页面 | 页面可用，空态/错态可处理，复制安装命令可用 |
| M7 | 可观测性与联调验收（指标/日志/告警/E2E） | 跨端 | P0 | S3（收口） | M1~M6 | Dashboard、告警规则、E2E 报告 | 关键链路可追踪；7 天灰度监控可执行 |

## 2) 推荐需要多少个员工（worker）及角色

| 方案 | 推荐人数 | 角色配置 | 适用场景 | 风险 |
|---|---:|---|---|---|
| 推荐方案（建议） | 6 | 1×PM/交付负责人，1×CLI工程师，2×后端工程师，1×前端工程师，1×QA/可观测性工程师 | 目标 5~7 周完成 P0 并进入 Beta | 风险最低，跨端依赖可控 |
| 压缩方案（最小） | 4 | 1×PM/Tech Lead，1×全栈A（CLI+Ingest），1×全栈B（Normalize+Query+前端），1×QA兼职 | 人力受限必须压缩时 | 串行增多，里程碑滑动概率高 |

### 推荐 Worker 清单（6人）

| Worker ID | 角色 | 主责模块 |
|---|---|---|
| `worker-1` | PM/交付负责人 | M0、里程碑推进、跨团队依赖管理 |
| `worker-2` | CLI 集成工程师（Go） | M1 |
| `worker-3` | 后端接入工程师（API/DB） | M2、M3 |
| `worker-4` | 后端数据管道工程师（Worker/Query） | M4、M5 |
| `worker-5` | 前端工程师（Next.js） | M6 |
| `worker-6` | QA/可观测性工程师 | M7 |

## 3) 每个 Worker 的任务边界、输入输出、依赖关系、验收标准

| Worker | 任务边界（负责） | 非边界（不负责） | 输入 | 输出 | 依赖关系 | 验收标准 |
|---|---|---|---|---|---|---|
| `worker-1` PM/交付负责人 | 冻结范围、排期、风险闭环、质量门禁 | 不写业务代码 | PRD、design、团队容量 | 里程碑计划、依赖清单、风险台账、周报 | 上游：无；下游：全员 | 每周里程碑按时评审；阻塞项 24h 内有 owner 与处置 |
| `worker-2` CLI 集成工程师 | `role-repo find` 后异步上报、重试与日志、不阻塞主流程 | 不负责 role-hub Web/API 页面 | M0 API 契约、密钥配置策略 | CLI 上报代码、单测、回归报告 | 依赖 `worker-3` 提供 ingest 契约与联调环境；被 `worker-6` E2E 使用 | Ingest 不可用时 `find` 仍成功；新增单测覆盖关键分支 |
| `worker-3` 后端接入工程师 | Ingest API、验签、幂等写入、DB 迁移与索引 | 不负责目录页面实现 | M0 契约、字段模型 | API 实现、迁移脚本、审计日志、接口文档 | 上游：`worker-1`；下游：`worker-4/5/6` | 重放请求被拒绝或幂等处理；迁移可执行可回滚 |
| `worker-4` 后端数据管道工程师 | Normalize/sync 任务、状态流转、Query API（列表/搜索/筛选/详情） | 不负责 CLI 修改 | M3 表结构、M2 数据输入 | worker 任务、Query API、重试策略 | 依赖 `worker-3` 数据模型；支撑 `worker-5` 页面与 `worker-6` 压测 | 默认仅返回 `verified`；P95 性能达 PRD 目标 |
| `worker-5` 前端工程师 | 首页/目录/角色详情/仓库详情；搜索筛选与空态错态 | 不负责后端状态机与迁移 | M0 契约、M5 Mock/真实 API | 可部署前端页面、组件测试、交互验收记录 | 早期依赖 Mock，联调依赖 `worker-4` | 4 个 P0 页面可用；复制安装命令可操作；关键路径 E2E 通过 |
| `worker-6` QA/可观测性工程师 | 指标、日志、告警、E2E 场景、发布门禁 | 不新增业务需求 | M1~M6 构建产物、监控平台权限 | 监控面板、告警规则、E2E/回归报告、发布检查单 | 横向依赖所有 worker | 链路含 request_id；核心告警可触发；发布检查单通过 |

## 4) 分配顺序与里程碑

| 顺序 | 里程碑 | 时间建议 | Worker 分配顺序 | 里程碑出口（验收） | 关键依赖 |
|---|---|---|---|---|---|
| 1 | M0 需求与契约冻结 | Week 1 | `worker-1` 主导，`worker-2/3/4/5/6` 参与评审 | PRD 范围冻结；API/DB/状态机版本 `v1` 发布 | 无 |
| 2 | M1 接入链路打通 | Week 2-3 | `worker-3` 与 `worker-2` 并行，`worker-6` 介入验证 | CLI 非阻塞上报；Ingest 成功率 >95%；幂等通过 | M0 |
| 3 | M2 数据标准化与目录查询 | Week 3-4 | `worker-4` 主导，`worker-3` 支持 | 状态流转稳定；Query API 提供 P0 能力 | M1、M3 |
| 4 | M3 前端可用与全链路联调 | Week 4-5 | `worker-5` 主导，`worker-4/6` 并行配合 | 4 个 P0 页面上线内测；仅 `verified` 可见 | M2 |
| 5 | M4 Beta 发布与观测收口 | Week 6-7 | `worker-6` 主导，`worker-1` 做 Go/No-Go | 7 天无 P0 故障；核心指标稳定 | M3 |

## 6) 风险与缓释（执行期）

| 风险 | 等级 | 触发信号 | 缓释动作 | Owner |
|---|---|---|---|---|
| GitHub API 限流导致 `verified` 时延超标 | 高 | Normalize 队列积压、失败率上升 | 指数退避、限流预算、失败重试分层 | `worker-4` |
| 跨仓联调导致节奏错位（CLI 仓 + role-hub 仓） | 高 | 联调窗口延迟 >2 天 | 契约先行、Mock 优先、每日对齐 15 分钟 | `worker-1` |
| 搜索查询性能不达 P95 目标 | 中 | 压测超阈值 | 索引补强、查询计划优化、分页约束 | `worker-3/4` |
| 观测缺失导致问题难定位 | 中 | 故障定位超过 1 小时 | request_id 全链路、最小告警基线先上线 | `worker-6` |

## 7) 证据等级标注（关键结论）

| 结论 | 证据等级 | 说明 |
|---|---|---|
| 采用 6 人推荐编组可在 5~7 周完成 P0-Beta | Assumption | 基于 PRD 复杂度与常见团队吞吐估算，需以实际 velocity 校准 |
| `M1/M2/M3` 与 `M6` 可并行是最高性价比路径 | Research | 来自现有 PRD 的“契约先行 + 非阻塞接入 + P0 页面范围”结构 |
| 先冻结契约（M0）可显著降低跨仓返工 | Research | 来自当前 design 中的 API 契约与异步链路约束 |

---

**最终建议（一句话）**：采用 6 人推荐方案，以 “M0 契约冻结 → 后端接入并行 → 前后端联调 → 观测收口” 的顺序推进，可在既定 PRD 约束下兼顾速度与可控性。
