# AI 团队流程脑暴细化（公共规则优先级版）

- 日期：2026-03-20
- 角色：general strategist
- 类型：流程设计脑暴（已审批版）
- 主题：模块拆分下的规则层级治理（公共规则 > 模块规则 > 任务规则）

---

## 1) 问题陈述与目标

### 问题陈述
在模块化推进中，若模块规则可反向影响公共规则，会导致：
1. 上下文基线不稳定，执行口径随任务漂移。
2. 同类任务跨模块结果不一致，验收标准失真。
3. 规则冲突难定位，返工成本上升。

### 目标
1. 明确固定规则层级：公共规则最高、模块规则次级、任务规则最低。
2. 建立“不可反向覆盖”的约束，避免模块阶段改写公共基线。
3. 将校验从“字段齐全”收敛为“声明-实体一致性”，降低无效阻断。
4. 在准确性优先策略下维持“模块验收标准化 + 全局人工验收”。

---

## 2) 约束与假设

### 约束
- 模块化执行阶段不得修改或影响公共规则。
- 公共规则冲突时，模块流转必须先阻断再分流。
- 当前阶段继续采用人工全局验收，不推进纯自动化全局门禁。

### 假设
- 已有 Root/Index/Module/Task/Evidence 的文档化结构。
- 系统可为规则和文档分配稳定 DocID。
- 负责人可承担公共规则变更审批职责。

---

## 3) 候选方案与取舍

### 方案 A：强治理分层（最终选择）
- 公共规则只读，模块规则仅继承与补充，任务文档强制双引用。
- 冲突即阻断，公共规则变更走独立流程。

**优点**：上下文稳定性最高，偏差最小，可追溯性最强。
**代价**：公共规则变更节奏较慢。

### 方案 B：冻结窗口分层
- 公共规则按周期冻结，窗口内模块仅适配不可改基线。

**优点**：吞吐较高。
**代价**：窗口期内问题可能延后处理。

### 方案 C：双轨规则（稳定轨/实验轨）
- 稳定轨严格执行公共规则，实验轨允许试验模块规则。

**优点**：探索能力更强。
**代价**：治理复杂度与维护成本上升。

---

## 4) 推荐设计（已确认）

### 4.1 Architecture
- 规则层级：**Public Rule > Module Rule > Task Rule**。
- 模块与任务均不得反向覆盖公共规则。
- 全流程沿用文档化记忆与索引实时更新机制。

### 4.2 Components
1. **Public Rule Registry**：维护公共规则版本与状态（draft/active/deprecated）。
2. **Module Rule Registry**：模块规则必须绑定 PublicRuleID，并声明继承关系。
3. **Reference Guard**：校验任务双引用（PublicRuleID + ModuleRuleID）。
4. **Override Blocker**：检测模块对公共规则的覆盖企图并阻断。
5. **Change Control Queue**：公共规则变更独立提案/评审/签署队列。
6. **Evidence Binder**：验收结论绑定 EvidenceRef。

### 4.3 Data Flow
1. 需求入口：创建 Root Doc，绑定当前 active PublicRuleID。
2. 模块建模：创建 Module Rule Doc，生成 ModuleRuleID。
3. 任务下发：Task Mini-Doc 强制双引用后进入执行。
4. Worker 执行：按任务回填输出与证据，不修改公共规则。
5. 模块验收：校验规则遵循与证据完整性（按声明项）。
6. 冲突分流：若与公共规则冲突，阻断模块并转公共规则变更队列。
7. 全局人工验收：业务闭环/关键路径/风险综合判断。

### 4.4 Error Handling（按本轮确认口径）
- 核心硬阻断：**declared_reference_not_found**（已声明引用但目标实体不存在）。
- 非阻断：未声明字段或缺少某部分说明，默认视为不适用。
- 冲突处理：模块规则覆盖公共规则时，冻结模块流转并进入变更审批。
- 回退策略：仅回退到最近检查点，不全链路重跑。

### 4.5 Testing（流程验证）
1. **声明-实体一致性校验**：仅校验已声明引用是否可解析。
2. **规则优先级校验**：模块规则不得覆盖公共规则。
3. **任务可执行性校验**：任务声明的引用必须全部存在。
4. **模块验收抽样**：高风险路径的声明证据是否有效。
5. **全局人工验收清单**：业务闭环、关键路径、风险可接受性。

---

## 5) 风险与缓解

1. **公共规则变更队列拥堵**
   - 缓解：公共规则变更与模块执行流解耦，设定快速审批 SLA。

2. **模块方误把“未声明”当规避手段**
   - 缓解：对关键任务模板保留最小声明集（目标、引用、验收结论）。

3. **规则冲突频繁触发阻断**
   - 缓解：在模块建模阶段提前做规则差异预检，减少执行期冲突。

4. **人工全局验收主观波动**
   - 缓解：固定全局验收清单与判定记录格式，保证可追溯。

---

## 6) 验证指标（2~3 周试点）

### 效率指标
- 冲突任务平均阻断时长
- 公共规则变更平均审批时长
- 模块任务端到端交付周期

### 质量指标
- 规则冲突率（每模块）
- 模块一次通过率
- 集成阶段关键缺陷发现率

### 治理指标
- 双引用合规率（任务）
- declared_reference_not_found 发生率
- 索引实时更新率

---

## 7) 本轮结论

1. 采用 **方案 A（强治理分层）** 作为默认机制。
2. 明确规则权重：**公共规则最高，其次模块规则，最后任务规则**。
3. 明确边界：**模块化阶段不得改写或影响公共规则**。
4. 校验策略改为“**只阻断已声明引用缺失**”，未声明项不作硬阻断。
5. 继续执行“模块标准化验收 + 全局人工验收”的准确性优先闭环。
