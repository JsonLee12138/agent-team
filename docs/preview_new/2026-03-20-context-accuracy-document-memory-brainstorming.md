# AI 团队流程脑暴细化（主文档：文档化记忆与规则层级治理）

- 日期：2026-03-20
- 角色：general strategist
- 类型：流程设计脑暴（合并确认版）
- 主题：降低思维偏差，保障上下文准确性，以文档化记忆与规则层级门禁形成闭环

---

## 1) 问题陈述与目标

### 问题陈述
当前多智能体协作中的核心风险不是能力不足，而是上下文漂移与规则漂移：
1. 依赖个体自主记忆，信息在多轮传递中被重写或遗漏。
2. 文档粒度过大、更新不及时，导致真实状态与记录状态错位。
3. 模块化推进中若可反向影响公共规则，会造成基线不稳定与验收失真。

### 目标
1. 将上下文基座从“自主记忆”切换为“文档化渐进记录”。
2. 建立稳定规则层级：**公共规则 > 模块规则 > 任务规则**。
3. 通过实时索引与引用校验，确保每一步可追溯、可校验、可回退。
4. 在准确性优先下持续执行“模块标准化验收 + 全局人工验收”。

---

## 2) 约束与原则

### 约束
- 流程按 TDD 验收标准前置：先标准，后开发。
- 当前阶段准确性优先，不推进纯自动化全局验收。
- 模块化执行阶段不得修改或影响公共规则。
- 文档必须短小、细分、可增量更新。

### 原则
- **No-Doc-No-Work**：无 DocID 不开工。
- **Index-First**：新增/变更文档必须实时更新索引。
- **Public-First**：公共规则权重最高，模块与任务不得反向覆盖。
- **Declared-Ref-Strict**：仅对“已声明引用缺失实体”做硬阻断。
- **No-Owner-Signoff-No-Exit**：无负责人签署不出模块。

---

## 3) 候选方案与取舍

### 方案 A：强治理分层（最终选择）
- 公共规则只读，模块规则仅继承与补充，任务文档强制双引用。
- 冲突即阻断，公共规则变更走独立流程。

**优点**：上下文稳定性最高、偏差最小、可追溯性最强。
**代价**：公共规则变更节奏较慢。

### 方案 B：分级强约束
- P0/P1 强约束，P2 中约束。

**优点**：吞吐较高。
**代价**：低优先级任务仍可能发生漂移。

### 方案 C：负责人集中编排
- 执行者只产出卡片，负责人集中整理索引和阶段文档。

**优点**：执行层负担小。
**代价**：时延高，负责人成为瓶颈。

## 4) 推荐设计（已确认）

### 4.1 Architecture
- 主体采用“强约束文档化闭环 + 规则层级门禁”。
- 规则层级：**Public Rule > Module Rule > Task Rule**。
- 角色分层：Worker（产出）→ Module Lead（模块门禁）→ Global Reviewer（全局人工验收）。
- 文档拓扑：
  - Root/Index Doc（全局入口）
  - Public Rule Doc（公共基线）
  - Module Rule Doc（模块边界与验收标准）
  - Task Mini-Doc（细粒度执行单元）
  - Evidence（测试/输出/人工判定依据）

### 4.2 Components
1. **Index Registry**：维护 DocID 关系图与阶段状态。
2. **Public Rule Registry**：维护公共规则版本（draft/active/deprecated）。
3. **Module Rule Registry**：模块规则绑定 PublicRuleID，并声明继承关系。
4. **Reference Guard**：校验任务双引用（PublicRuleID + ModuleRuleID）。
5. **Override Blocker**：检测并阻断模块对公共规则的覆盖企图。
6. **Owner Review Queue**：负责人审阅、签署、退回。
7. **Evidence Binder**：将验收结论绑定 EvidenceRef。
8. **Deviation Ledger**：沉淀偏差类型与修复动作。
9. **Change Control Queue**：公共规则变更独立提案/评审/签署队列。

### 4.3 Data Flow
1. 需求入口：创建 Root Doc + 全局验收标准，绑定 active PublicRuleID。
2. 模块拆解：创建 Module Rule Doc + 模块验收标准，生成 ModuleRuleID。
3. 任务下发：创建 Task Mini-Doc，强制双引用后进入执行。
4. Worker 执行与自检：增量更新 mini-doc，绑定 EvidenceRef。
5. 模块验收：校验公共规则遵循、模块契约一致、声明证据有效。
6. 冲突分流：若模块规则与公共规则冲突，阻断模块并进入公共规则变更流程。
7. 集成冒烟：仅验证跨模块关键路径。
8. 全局人工验收：业务闭环综合判断并给出发布结论。

### 4.4 Error Handling（合并后口径）
- **硬阻断 A：declared_reference_not_found**
  - 已声明引用（DocID/EvidenceRef/RuleRef）但目标实体不存在。
- **硬阻断 B：rule_override_conflict**
  - 模块规则试图覆盖公共规则。
- **非阻断**
  - 未声明字段、缺少某部分说明：默认视为不适用。
- 回退策略：仅回退最近检查点，不全链路重跑。

### 4.5 Testing（流程验证）
1. **声明-实体一致性校验**：仅校验已声明引用是否可解析。
2. **规则优先级校验**：模块规则不得覆盖公共规则。
3. **任务可执行性校验**：任务已声明引用必须全部存在。
4. **模块验收抽样**：高风险路径的声明证据有效性。
5. **全局人工验收清单**：业务闭环、关键路径、风险可接受性。

## 5) 风险与缓解

1. **负责人审阅/审批成为瓶颈**
   - 缓解：负责人只审契约一致性、规则冲突与偏差，不审执行细节。

2. **小文档数量激增**
   - 缓解：模板化 + 字段上限，禁止长篇扩写。

3. **索引实时更新被跳过**
   - 缓解：设为硬门禁，不更新即阻断流转。

4. **公共规则变更队列拥堵**
   - 缓解：公共规则变更与模块执行流解耦，设定审批 SLA。

5. **“未声明”被误用为规避手段**
   - 缓解：关键任务模板保留最小声明集（目标、引用、验收结论）。

---

## 6) 验证指标（2~3 周试点）

### 效率指标
- 单任务平均审阅时长
- 冲突任务平均阻断时长
- 公共规则变更平均审批时长
- 端到端交付周期

### 质量指标
- 任务偏差率（目标与交付不一致）
- 模块一次通过率
- 规则冲突率（每模块）
- 集成阶段关键缺陷发现率

### 文档治理指标
- 索引实时更新率
- 双引用合规率（任务）
- declared_reference_not_found 发生率
- 证据绑定完整率（按声明项）
